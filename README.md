# Continuation

A personal journal reader for Christine Clay Moreau. Journal entries live in Google Drive — the app reads them and renders them as a styled, year-themed archive.

## How it works

There is no CMS or admin interface. Content is managed entirely in Google Drive:

- A single Drive folder contains files named by date (`2008-11-14`, `March 4, 2009`, etc.)
- The app groups files by year, parses dates into URL slugs, and exports each doc as HTML on request
- Drive content is cached for 1 hour (file list) and per-file (entry HTML)

Entries are accessed at `/:year/:slug`, e.g. `/2008/nov-14-2008`.

## Architecture

| Concern | Approach |
|---|---|
| Content storage | Google Drive (read-only, service account) |
| Database | SQLite via Solid Cache / Queue / Cable |
| Deployment | Fly.io (single machine, 1GB, `ord` region) |
| Asset pipeline | Propshaft + importmap |
| Background jobs | Solid Queue |

### Notable features

**Year-themed styling** — each year gets its own visual treatment applied via `data-year` on the body. 2008 is MySpace/emo (hot pink glows, Impact font, dark background). 2009 is cosmic/neon nebula.

**AI-generated comments** — entries show era-appropriate fake blog comments generated by Claude (Haiku). Comments are generated once and stored in SQLite (`generated_comments` table). Requires `ANTHROPIC_API_KEY`. If the key is absent the comments section is silently skipped.

**Era-appropriate banner ads** — configured in `config/banner_ads.yml`, rendered per-year. Purely cosmetic.

**Visitor counter** — stored in SQLite (`visitor_counters` table), incremented on each homepage load.

## Environment variables

| Variable | Description |
|---|---|
| `GOOGLE_DRIVE_FOLDER_ID` | ID of the Drive folder containing journal files |
| `GOOGLE_SERVICE_ACCOUNT_JSON` | Full service account JSON as a single-line string |
| `ANTHROPIC_API_KEY` | Enables AI comment generation (optional) |

See `.env.example`. In development, copy it to `.env` and fill in values.

The Google service account needs **Viewer** access to the Drive folder. No write permissions are used.

## Local setup

```bash
bundle install
bin/rails db:prepare
bin/dev
```

Requires a `.env` file with the variables above. Without `GOOGLE_DRIVE_FOLDER_ID` and `GOOGLE_SERVICE_ACCOUNT_JSON` the app will error on any page load.

## Testing

```bash
bin/rails test
```

Security checks (run in CI):

```bash
bin/brakeman --no-pager   # static analysis
bin/bundler-audit         # known CVEs in gems
bin/importmap audit       # JS dependency audit
```

## Deployment

Deployed to Fly.io. Pushes to `main` deploy automatically via GitHub Actions.

```bash
fly deploy          # manual deploy
fly logs            # tail logs
fly ssh console     # open a console on the running machine
bin/rails console   # shortcut defined in fly.toml
```

SQLite data is stored on a persistent Fly volume (`/data`). The volume auto-extends at 80% capacity up to 10GB.

## CI / dependency automation

**CI** runs on every PR and push to `main`: Brakeman, bundler-audit, importmap audit, RuboCop, and the test suite.

**Dependabot** opens weekly PRs for gem and GitHub Actions updates. These are automatically approved and squash-merged once CI passes.

**Weekly dependency update** (`Update Ruby and dependencies` workflow, Mondays 9am UTC) — checks for CVEs in current gems, updates `.ruby-version` to the latest stable Ruby, runs `bundle update --all`, re-checks for CVEs, then opens and immediately merges a PR if anything changed.

The `/update-deps` Claude Code skill runs the same update process locally.
